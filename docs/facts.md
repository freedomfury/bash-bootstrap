# Facts Generators

Two scripts in `bin/` gather system facts and write sourceable `.env` files to
`tmp/cache/facts/`. Each script uses a different data structure:
`facts-global-os` uses standard scalar variables, while `facts-cloud-init` 
uses a single associative array. Both are self-contained and require no 
pre-declaration before sourcing.

---

## Shell Compatibility

> [!IMPORTANT]
> Facts files generated by these scripts are **Bash 5.3+ compatible only**. 
> They follow the project's strict architectural distinction for `.env` files:
> - **Must always carry the `.env` extension** to signal their Bash-specific nature.
> - **Must only be sourced by Bash scripts** (Type 2 or Type 3).
> - **Must never be sourced by POSIX `/bin/sh` scripts** (`vendor/build`, `vendor/clean`).
>
> This is because they may use advanced Bash features (like associative arrays
> or shell-safe quoting) that are not POSIX-compliant.

## `bin/facts-global-os`

Collects OS and hardware facts from the local system at runtime.

**Output:** `tmp/cache/facts/facts-global-os.env`
**Default prefix:** `MKS_OS_FACTS`

```bash
bin/facts-global-os [--prefix <name>] [--output <file>] [--verbose]
```

**Sources:** `/etc/os-release`, `uname`, `hostname`, `ip route`, `nproc`,
`/proc/meminfo`, `systemd-detect-virt`, `timedatectl`.

### Key variables

| Variable | Description |
|---|---|
| `MKS_OS_FACTS_PRETTY_NAME` | Human-readable OS name, e.g. `Ubuntu 22.04.3 LTS` |
| `MKS_OS_FACTS_ID` | Distro ID, e.g. `ubuntu` |
| `MKS_OS_FACTS_VERSION_ID` | Version string, e.g. `22.04` |
| `MKS_OS_FACTS_ARCH` | CPU architecture, e.g. `x86_64` |
| `MKS_OS_FACTS_HOSTNAME_SHORT` | Short hostname |
| `MKS_OS_FACTS_HOSTNAME_FQDN` | Fully qualified domain name |
| `MKS_OS_FACTS_IPV4` | Primary IPv4 address |
| `MKS_OS_FACTS_KERNEL_RELEASE` | Kernel version string |
| `MKS_OS_FACTS_CPU_COUNT` | Number of logical CPUs |
| `MKS_OS_FACTS_MEM_TOTAL_KB` | Total memory in kilobytes |
| `MKS_OS_FACTS_NIC_DEFAULT` | Default network interface name |
| `MKS_OS_FACTS_GATEWAY_IPV4` | Default gateway IP address |
| `MKS_OS_FACTS_VIRT_TYPE` | Virtualisation type (`kvm`, `xen`, `none`, etc.) |
| `MKS_OS_FACTS_TIMEZONE` | IANA timezone, e.g. `America/New_York` |
| `MKS_OS_FACTS_TIMEZONE_ABBR` | Timezone abbreviation, e.g. `EST` |
| `MKS_OS_FACTS_TIME` | ISO-8601 timestamp when the file was generated |
| `MKS_OS_FACTS_EPOCH` | Unix epoch when the file was generated |

### Usage in a script

```bash
source "tmp/cache/facts/facts-global-os.env"
printf 'Running on %s (%s)\n' "${MKS_OS_FACTS_PRETTY_NAME}" "${MKS_OS_FACTS_ARCH}"
```

## `bin/facts-cloud-init`

Parses a cloud-init `instance-data.json` file and flattens all scalar values
into an associative array using dot-notation keys.

**Output:** `tmp/cache/facts/facts-cloud-init.env`
**Default prefix:** `MKS_CI_FACTS`

```bash
bin/facts-cloud-init --input <file> [--prefix <name>] [--output <file>] [--verbose]
```

`--input` is required. To generate the JSON on a live cloud instance:

```bash
sudo cloud-init query -a > tmp/instance-data.json
bin/facts-cloud-init --input tmp/instance-data.json
```

### Key variables

Keys mirror the cloud-init `instance-data.json` structure with `.` as the
separator. The set of keys varies by cloud provider; these are common ones:

| Variable | Description |
|---|---|
| `MKS_CI_FACTS["v1.cloud_name"]` | Cloud provider name, e.g. `aws`, `gce`, `azure` |
| `MKS_CI_FACTS["v1.distro"]` | Distro ID, e.g. `ubuntu` |
| `MKS_CI_FACTS["v1.distro_release"]` | Distro release name, e.g. `jammy` |
| `MKS_CI_FACTS["v1.distro_version"]` | Distro version, e.g. `22.04` |
| `MKS_CI_FACTS["v1.instance_id"]` | Cloud instance ID |
| `MKS_CI_FACTS["v1.local_hostname"]` | Short hostname assigned by the cloud |
| `MKS_CI_FACTS["v1.machine"]` | CPU architecture, e.g. `x86_64` |
| `MKS_CI_FACTS["v1.platform"]` | Cloud platform, e.g. `ec2` |
| `MKS_CI_FACTS["v1.public_ipv4"]` | Public IPv4 address |
| `MKS_CI_FACTS["v1.region"]` | Cloud region, e.g. `us-east-2` |
| `MKS_CI_FACTS["v1.availability_zone"]` | Availability zone |
| `MKS_CI_FACTS["ds.meta_data.instance-type"]` | Instance type, e.g. `t3.medium` |
| `MKS_CI_FACTS["ds.meta_data.ami-id"]` | AMI or image ID (AWS) |
| `MKS_CI_FACTS_EPOCH` | Unix epoch when the file was generated |

### Usage in a script

```bash
source "tmp/cache/facts/facts-cloud-init.env"
printf 'Cloud: %s  Region: %s  Instance: %s\n' \
  "${MKS_CI_FACTS[v1.cloud_name]}" \
  "${MKS_CI_FACTS[v1.region]}" \
  "${MKS_CI_FACTS[v1.instance_id]}"
```

## TTL and cache invalidation

Both scripts write an `_EPOCH` variable (e.g. `MKS_OS_FACTS_EPOCH`,
`MKS_CI_FACTS_EPOCH`) recording the Unix timestamp at generation time.
Callers can compare this against `${EPOCHSECONDS}` to decide whether to
regenerate before sourcing:

```bash
source "tmp/cache/facts/facts-global-os.env"
if (( EPOCHSECONDS - MKS_OS_FACTS_EPOCH > 3600 )); then
  bin/facts-global-os  # regenerate — cache is older than 1 hour
  source "tmp/cache/facts/facts-global-os.env"
fi
```
