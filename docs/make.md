# Makefile Architecture

This document describes how the Make system works: its file layout, style
conventions, and the key GNU Make features it depends on.

---

## File Layout

```
Makefile            ← Entry point; includes inc/ fragments
inc/
  common.mk         ← Shell configuration and shared settings
bootstrap.d/
  main.mk           ← Bootstrap pipeline targets
inc/
  make-help         ← Parses #@ annotations to generate the help screen
```

The root `Makefile` is kept intentionally short. It delegates shared Make
configuration to `inc/common.mk` and subproject pipelines to per-subproject
include files (e.g. `bootstrap.d/main.mk`). Adding a new subproject means
adding one `<subproject>/main.mk` file and one `include` line.

---

## Shell Configuration (`inc/common.mk`)

```makefile
SHELL        := /usr/bin/env mks-bash
.SHELLFLAGS  := -e -c
.ONESHELL:
.DELETE_ON_ERROR:
```

**`SHELL` and `.SHELLFLAGS`** — The Makefile uses `mks-bash` (the vendored Bash executable) as its shell. This ensures that Makefile recipes can only run when the vendor environment has been properly set up via `source exports`.

> **Note**: The actual `inc/common.mk` uses `$(CURDIR)/vendor/bin/mks-bash` (an absolute path) rather than `/usr/bin/env mks-bash`. This is because GNU Make's `SHELL` variable doesn't support `/usr/bin/env` lookup reliably. The absolute path ensures the correct binary is used. 

| Flag | Effect |
|---|---|
| `-e` | Exit on error |
| `-c` | Required by Make: execute the recipe string |

**`.ONESHELL:`** — Without this, each line of a multi-line recipe runs in
its own shell process. With it, the entire recipe runs in a single shell.
This is what makes `source exports` at the top of a recipe useful — without
`.ONESHELL:`, the sourced environment would vanish before the next line ran.

**`.DELETE_ON_ERROR:`** — If a recipe fails, Make deletes the target file.
This prevents stale build artifacts from making subsequent runs appear
successful when they were not.

**`.DEFAULT_GOAL := help`** — Running `make` with no arguments prints the
help screen rather than executing the first target in the file.

## Vendor Sentinel (`vendor.lock`)

`vendor.lock` is a sentinel file created by `./mks-vendor` (which internally runs `vendor/build`). It indicates that the statically-compiled toolkit has been built and is available in `vendor/bin/`.

The root `Makefile` depends on this file but **does not provide a recipe to build it**. If the file is missing, `make` will fail, directing the developer to run the manual setup:

```bash
./mks-vendor
```

This decoupling ensures that the tools used by the Makefile are not accidentally deleted or modified during a standard `make clean`.

## `exports` File

The `exports` file is a **static** file (checked into version control) that defines the core project environment. It is not generated by Make.

When sourced, `exports` sets:

```bash
MKS_ROOT          # Absolute path to project root
MKS_TMP           # Temporary directory path
MKS_VENDOR        # Path to vendor binaries
MKS_BASH          # Path to mks-bash executable
MKS_EXPORTED      # Flag indicating exports was sourced
PATH              # Vendor binaries prepended
```

All variables are exported and marked readonly using `declare -rx`. Every script guards against a missing `MKS_ROOT` as its first act.

### `exports` vs `.env` Files

Both `exports` and `.env` files are **Bash 5.3+ compatible**. The distinction is semantic:

- **`exports` Files**: Define the mandatory environment for a project or subproject. Always sourced first. The root `exports` is static and checked in. Subprojects may have their own `exports` files sourced after the root.
- **`.env` Files**: Contain situational data or configuration. These are exclusively sourced by downstream Bash scripts and task runners, never by `make` recipes.

## Help System

The `.DEFAULT_GOAL` is `help`. The help target invokes `inc/make-help`, which
parses all Makefile fragments passed via `$(MAKEFILE_LIST)`:

```makefile
help:
	$(CURDIR)/inc/make-help $(MAKEFILE_LIST)
```

Targets are documented with `#@` annotations on the line immediately above:

```makefile
#@ Run the full bootstrap subproject pipeline.
bootstrap: depends
```

`inc/make-help` reads each file, collects `#@` lines as descriptions, and
pairs them with the next non-comment, non-blank target line. Output is
formatted as a two-column table with color.

Targets without a `#@` annotation are excluded from help output. This is
the sole mechanism for controlling visibility — no naming convention is
required. To hide a target from `make help`, simply omit the `#@` line.

## Bootstrap Pipeline (`bootstrap.d/main.mk`)

### Two-Phase Invocation

The `bootstrap` target uses a two-phase pattern:

```makefile
bootstrap: depends
	@echo "==- bootstrap -=="
	$(MAKE) --no-print-directory -f Makefile bootstrapd
```

Phase 1 ensures all dependencies (vendor toolkit) are available via
`depends`. Phase 2 invokes a recursive `$(MAKE)` call targeting `bootstrapd`.
The recursive Make inherits the current environment, but each recipe
in `bootstrap.d/main.mk` also sources `exports` directly to ensure
`MKS_ROOT` and siblings are always available regardless of how the
target is invoked.

### Linear Pipeline via Dependency Chaining

The bootstrap pipeline is expressed as a dependency chain:

```makefile
bootstrapd: scoreboard
scoreboard: tasks
tasks: preflight
```

This reads bottom-up: `preflight` must complete before `tasks`, which
must complete before `scoreboard`. Make enforces the ordering. Adding a new
stage means inserting it into the chain — no explicit sequencing code needed.

### Target Visibility

Whether a target appears in `make help` is controlled entirely by the
presence of a `#@` annotation. Targets without one are excluded from help
output regardless of their name. No naming prefix is required to hide a
target — omitting the `#@` line is sufficient.

## Maintenance Targets

```makefile
#@ Clean all temporary files from the tmp directory.
clean:
	@echo "==- clean -=="
	rm -vrf tmp/*
```

`clean` removes project-level generated artifacts: everything inside `tmp/`.
Note: `exports` is a static file and is not removed by `clean`.

Important: `clean` **does not** remove the vendored toolkit or the
`vendor.lock` file. This decoupling allows you to clean the project
state without losing the tools needed to build it. To wipe the toolkit,
run the standalone cleanup script:

```bash
./vendor/clean
```

`clean` is not declared `.PHONY` — it is unconditional because it has no
prerequisites and Make has nothing to compare it against. Omitting `.PHONY`
here is harmless: Make will always run the recipe.

## Style Conventions

| Convention | Value |
|---|---|
| Indentation | Hard tab (required by Make) |
| Include fragments | `<subproject>/main.mk` — one file per subproject |
| Public target documentation | `#@` annotation on the preceding line |
| Internal targets | No `#@` annotation (omitting it hides the target from help) |
| Progress output | `@echo "==- <target> -=="` for user-visible targets; `@echo "==> <target>"` for steps within a target |
| Phony declaration | `.PHONY:` line immediately before the first target in logical groupings |
